# Product Brief: ai product manager

## One-Sentence Definition

ai product manager is an AI product manager that turns rough software ideas, design direction, architecture constraints, and team context into Scrum-ready development guidance.

## Product Intent

Software teams often fail not because developers cannot code, but because the development input is incomplete:

- product goals are vague
- user stories are missing
- acceptance criteria are unclear
- design and architecture constraints are not connected
- developers do not know what should be built first
- sprint scope changes without visible tradeoffs

ai product manager exists to flatten that ambiguity before development starts.

## Target Users

Primary users:

- product managers
- startup founders
- design leads
- tech leads
- small development teams
- solo builders using AI coding tools

Secondary users:

- QA testers
- UI/UX designers
- solution architects
- scrum masters

## Core Value

The product gives a team a clear Scrum development package:

- product brief
- clarified scope
- user stories
- acceptance criteria
- prioritized backlog
- sprint plan
- developer guidance
- QA checklist

The goal is to help developers build the right thing with fewer meetings, fewer vague tickets, and less rework.

## MVP Positioning

The first MVP is Scrum-only.

It should not try to become a full project-management suite. It should focus on one valuable workflow:

1. User describes a software idea or feature.
2. AI asks focused clarification questions.
3. AI creates a product brief.
4. AI creates Scrum user stories.
5. AI creates acceptance criteria.
6. AI prioritizes the first backlog.
7. AI proposes a sprint goal and sprint backlog.
8. AI generates developer-facing guidance and QA checks.

## User Problem

Teams need a bridge between idea, design, architecture, and implementation.

Current tools usually separate these concerns:

- design tools hold UI decisions
- docs hold product ideas
- issue trackers hold tasks
- code repositories hold implementation
- AI chat holds temporary reasoning

The MVP should connect these into a single structured Scrum planning flow.

## MVP Success Criteria

The MVP is successful if a user can enter a rough product idea and receive a usable first sprint package that includes:

- a concise product brief
- 5 to 12 prioritized user stories
- acceptance criteria for each selected story
- a sprint goal
- a sprint backlog
- developer notes for implementation
- QA checklist for verification

The output should be specific enough that a developer can begin work without asking basic product questions.

## Non-Goals

The MVP does not include:

- full Jira, Linear, GitHub Projects, or Notion integration
- multi-framework project management
- enterprise permissions
- resource capacity forecasting
- automated code generation
- deployment automation
- long-term portfolio analytics
- team chat replacement

## Product Principles

1. Structured output over chat history.
   - AI reasoning must become usable artifacts.

2. Acceptance criteria before implementation.
   - Developers should not code from vague feature titles.

3. Sprint scope must be visible.
   - The system should show what is included, what is deferred, and why.

4. Design and architecture constraints must affect backlog decisions.
   - They should not live as disconnected notes.

5. Every output must help someone take action.
   - If the output cannot guide product, design, development, or QA work, it should be simplified or removed.

## Current Product Boundary

The product should behave like a focused Scrum product manager, not a generic chatbot.

The MVP should support:

- product clarification
- Scrum artifact generation
- backlog shaping
- sprint planning
- developer guidance
- QA verification support

The MVP should not support broader agile consulting until the Scrum flow is proven.
